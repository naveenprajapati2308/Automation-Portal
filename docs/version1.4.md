# Automation Portal – Version 1.4 Requirements

## Objective

The goal is to build a complete **Full Stack SDET Automation Portal** that acts as a centralized platform for managing automation execution, reports, analytics, and future project wise exicution have role and inside have same functionlity to  management.

Currently, the portal is being developed for **one automation project**, but the architecture must be designed so that it can easily support **multiple projects, multiple users, and role-based access control** in future versions.

---

# 1. Dashboard

The Dashboard is already on the right track.

It should display:

* Latest execution summary
* Previous execution statistics
* Total Executions
* Passed Tests
* Failed Tests
* Skipped Tests
* Pass Percentage
* Execution Duration
* Execution Trends
* Module-wise Analytics
* Charts and Graphs
* Recent Executions
* Environment Information
* Any other useful KPIs

The dashboard should not contain hardcoded values. Every component must fetch its data dynamically from the database after execution results are parsed.

# 4. Dynamic Execution Configuration

Currently, execution options should NOT be hardcoded.

Instead,

An Administrator should have a configuration screen where they can register new automation modules.

For every execution entry, the administrator should be able to configure:

* Module Name
* Project Name
* Suite Name
* Display Name
* Framework Path
* Runner Method/API
* XML File
* Description
* Active/Inactive Status

Once configured,

those entries should automatically appear inside the Execution Center dropdown.

This removes the need to change code whenever a new module is added.

---

# 5. Execution Process

When the user clicks **Run**:

The portal should

* Call the Portal API
* Trigger the Execution Server
* Execution Server should call the Framework Runner
* Framework Runner executes TestNG
* Automation starts

After execution finishes,

the framework generates

* Extent Report
* TestNG XML
* Logs
* Screenshots
* Other execution artifacts
* those are helpful for the dashboard and my automation portal 

The portal then parses all generated files and stores the extracted data into the database.

After parsing,

all portal pages should automatically display updated information.

# 16. Future Scope (Beyond Version 1.4)

Currently,

the portal supports only one automation project.

Future versions should support:

* Multiple Projects
* User Management
* Project-wise Access
* Role-Based Access Control (RBAC)
* Team Management
* Permissions
* Organization Support
* Multi-Tenant Architecture
* Scheduler
* Notifications
* CI/CD Integration
* API Automation
* Mobile Automation
* Performance Testing
* Cloud Execution
* Distributed Execution

The architecture should be designed today so these features can be added without major redesign.
