# Phase 6 Architecture Correction (Very Important)

After reviewing the implementation plan, I found that Phase 6 does not match my actual vision.

You have interpreted the Automation Portal as a system that waits for execution to finish, copies report files, parses them, stores the parsed data into the database, and finally displays the information on the dashboard.

This is **NOT** the architecture I want.

I am **not building a Report Viewer**.

I am building a **complete Automation Execution Platform**.

## Current Understanding (Incorrect)

Current implementation flow:

Automation Portal
→ Run Execution
→ Wait until framework finishes
→ Copy report files
→ Parse HTML/XML reports
→ Store parsed data into database
→ Show dashboard

This is a batch-processing architecture and is not what I want.

# Expected Architecture

The Automation Portal should become the central controller of the Automation Framework.

The framework should not simply generate reports.

Instead, it should continuously communicate with the portal while execution is running.

Expected flow:

Automation Portal
↓
Execution Center
↓
Execution API
↓
Framework Runner Service
↓
Launch TestNG Suite
↓
Framework executes tests
↓
Framework continuously sends execution events
↓
Backend receives events
↓
Stores data immediately
↓
Dashboard updates in real time

The dashboard should never wait for execution completion.

# Framework Runner

Currently I have a dummy runner server inside the automation framework.

Instead of using it as a temporary solution, redesign it into a proper Framework Runner Service.

The Runner should expose functionality to execute:

* Master Suite
* Module Suite
* Individual XML
* Individual Test Class
* Single Test Method
* Smoke
* Regression
* Sanity
* Custom Execution

The portal should simply call the Runner API.

---

# Framework Changes Required

Instead of generating reports only, modify the automation framework so that it continuously publishes execution events.

Whenever any TestNG listener event occurs, the framework should notify the backend.

Examples:

onExecutionStart
onSuiteStart
onTestStart
onTestSuccess
onTestFailure
onTestSkipped
onScreenshotCaptured
onLogGenerated
onModuleCompleted
onExecutionCompleted

These events should be pushed immediately to the backend.

Do not wait for execution completion.

---

# Backend Responsibilities

The backend should expose APIs specifically for execution events.

Instead of creating a parser that runs only after execution finishes, create an Execution Event Engine.

Responsibilities:

* Receive execution events
* Validate incoming data
* Store data into database
* Update execution status
* Update module statistics
* Update dashboard statistics
* Publish live updates to frontend

This should happen throughout execution.

# Report Parser

The Report Parser should NOT become the primary data source.

Reports are only final artifacts.

The parser should be used only to:

* Validate execution
* Extract additional metadata
* Read historical executions
* Import external reports
* Recover missing information if required

The dashboard should already contain the execution information because it was received during execution.
